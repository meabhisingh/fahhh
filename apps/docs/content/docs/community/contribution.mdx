# Contribution

Fahhh is still early, so contributions are most useful when they keep the framework small, typed, and easy to deploy.

## Create a Test App

Use the package manager you prefer:

<Tabs persist groupId="packageManager" items={["npm", "yarn", "pnpm", "bun"]}>
  <Tab value="npm">
    ```bash
    npx create-fahhh my-app
    ```
  </Tab>
  <Tab value="yarn">
    ```bash
    yarn create-fahhh my-app
    ```
  </Tab>
  <Tab value="pnpm">
    ```bash
    pnpm dlx create-fahhh my-app
    ```
  </Tab>
  <Tab value="bun">
    ```bash
    bunx dlx create-fahhh my-app
    ```
  </Tab>
</Tabs>

For framework development, work from the monorepo instead:

```bash
git clone https://github.com/meabhisingh/fahhh.git
cd fahhh
pnpm install
```

## Common Commands

<Tabs persist groupId="packageManager" items={["npm", "yarn", "pnpm", "bun"]}>
  <Tab value="npm">
    ```bash
    npm run build
    npm run typecheck
    ```
  </Tab>
  <Tab value="yarn">
    ```bash
    yarn build
    yarn typecheck
    ```
  </Tab>
  <Tab value="pnpm">
    ```bash
    pnpm build
    pnpm typecheck
    ```
  </Tab>
  <Tab value="bun">
    ```bash
    bun run build
    bun run typecheck
    ```
  </Tab>
</Tabs>

When you are only changing one package, filter the command:

```bash
pnpm --filter @fahhh/runtime build
pnpm --filter @fahhh/compiler build
pnpm --filter fahhh build
```

Test the example app when your change affects routing, typegen, dev, build, start, or deploy:

```bash
cd examples/basic
pnpm typecheck
pnpm build
pnpm start
```

## Generated Files

Do not commit generated output.

```text
.fahhh
dist
dist-server
node_modules
```

<Callout title="Why ignore .fahhh?">
  The `.fahhh` directory is generated by the framework. It can contain manifests,
  virtual API types, and deployment staging files.
</Callout>

## Commit Messages

Fahhh uses conventional commits. Commitlint should reject messages that do not follow this shape.

```bash
feat(runtime): add node production server
fix(cloudflare): keep runtime imports portable on Windows
docs: add contribution guide
test(core): cover build and start output
```

Use these types for most changes:

- `feat`
- `fix`
- `docs`
- `test`
- `refactor`
- `chore`
- `build`
- `ci`

## Git Hooks

Fahhh uses Lefthook for local checks.

```bash
pnpm lefthook install
```

Expected hook split:

- `pre-commit` runs fast formatting and lint checks.
- `commit-msg` runs Commitlint.
- `pre-push` runs typecheck and selected tests.

<Callout title="Keep hooks fast">
  Local hooks should catch obvious mistakes without making every commit painful.
  Full integration coverage belongs in CI.
</Callout>

## Pull Requests

Before opening a pull request, run:

```bash
pnpm typecheck
pnpm build
```

Include:

- What changed.
- Why it changed.
- How you tested it.
- Any adapter or platform assumptions.

## Contribution Areas

Good first areas:

- Docs examples.
- Error message polish.
- `create-fahhh` template fixes.
- Example app coverage.
- Tests for route matching and typegen.

Larger areas:

- Netlify adapter.
- Vercel adapter.
- AWS Lambda adapter.
- More integration tests for `fahhh dev`, `fahhh build`, and `fahhh start`.
